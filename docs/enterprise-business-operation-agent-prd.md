# PRD：企业办公 Agent - Dano

## 1. 背景

企业员工每天要在 OA、HR、财务、差旅、CRM、ERP、审批系统里完成大量重复性操作。典型场景包括请假、出差申请、报销、订票、用章、采购申请、审批查询、客户信息维护等。

这些操作的难点通常不是“不会判断”，而是：

- 要知道公司制度和流程。
- 要找到正确系统和入口。
- 要填写大量字段。
- 要上传材料。
- 要处理审批链、权限、校验和异常提示。
- 要在不同系统之间重复搬运信息。

传统解决方式有两类：

- 深度系统集成：效果稳定，但每个系统都要开发接口、适配权限、维护字段映射，交付慢、成本高。
- RPA/脚本自动化：可以模拟人工操作，但配置重、脆弱，业务人员难以直接使用，也很难处理非结构化输入。

大模型带来的机会是：用户可以用自然语言表达业务意图，Agent 可以理解制度、流程、字段和页面，再通过 API、MCP、企业连接器或浏览器自动化完成实际操作。

因此，本产品不是普通知识库问答，也不是单纯 RPA，而是面向企业业务系统的可执行个人业务助理。

## 2. 要解决的问题

### 2.1 员工侧问题

员工不想为了简单事务反复进入多个系统、找菜单、填表、查规则、复制信息。员工真正想要的是：

> 我说一句“帮我请明天下午半天年假”，系统能理解我要办什么，补齐必要信息，帮我打开对应系统填好表，并在提交前让我确认。

### 2.2 企业侧问题

企业希望降低内部事务处理成本，但不能牺牲安全、权限和现有业务流程。企业关心的是：

- Agent 是否越权。
- 页面或流程变化后是否会误操作。
- 是否每接一个系统都需要大量开发。

### 2.3 产品侧问题

产品要解决的核心矛盾是：

> 企业业务系统高度定制，但客户又希望低成本、快速接入、尽量少开发。

所以产品目标不是承诺“完全不用定制”，而是把传统定制开发变成低配置、可验证、可发布的业务 Skill 生产过程。

## 3. 产品定位

企业办公 Agent 是一个面向员工的个人业务助理，也是一个面向企业管理员的业务 Skill 管理平台。

一句话定位：

> 让员工不用手动进入 OA、HR、财务等系统填表，只需用自然语言说明要办的事，Agent 自动理解流程、补齐信息、代填系统，并在关键动作前请求确认。

更准确的产品形态：

- 前台是员工可用的自然语言业务助理。
- 中台是企业业务 Skill Factory。
- 底层是 API/连接器优先、Browser Worker 兜底的执行引擎。

产品不替代 OA 首页，也不要求 P0 阶段把 Agent 嵌入客户 OA。P0 默认入口是独立 Agent Workbench，OA、HR、财务等系统作为被连接的目标业务系统。员工在 Workbench 中发起业务，Agent 通过目标系统 API 或连接器完成操作，并返回目标系统生成的业务结果。

## 4. 目标用户

### 4.1 企业员工

需要频繁办理内部事务，希望用自然语言完成请假、出差、报销、审批查询等操作。

### 4.2 企业管理者

关心员工效率、流程正确性、审批质量和内部系统使用成本。

### 4.3 企业 IT / 数字化团队

负责接入统一认证、企业系统、权限和流程配置，希望减少重复开发和长期维护成本。

### 4.4 业务管理员

熟悉公司制度和流程，负责确认 Agent 生成的业务 Skill 是否符合实际业务规则。

## 5. 产品目标

### 5.1 核心目标

1. 员工可以用自然语言发起常见企业事务。
2. Agent 可以把自然语言转换成结构化业务任务。
3. Agent 可以基于制度、操作手册、表单结构和用户上下文补齐流程信息。
4. Agent 可以通过 API、连接器或浏览器自动化完成系统操作。
5. 高风险动作必须由用户确认后执行。
6. 新企业、新系统、新流程的接入成本要明显低于传统定制开发。

### 5.2 不追求的目标

1. 不做万能 Agent。
2. 不承诺零配置接入任意企业系统。
3. 不让模型无约束地自由点击生产系统。
4. 不在 P0 阶段覆盖所有 OA、财务、差旅、采购、CRM 场景。
5. 不把知识库问答当成最终产品形态。

## 6. 产品范围

### 6.1 要做什么

产品需要提供以下能力：

1. 自然语言任务入口：员工用聊天方式发起业务操作。
2. 业务意图识别：识别请假、出差、报销、审批查询、用章、采购等任务类型。
3. 字段抽取与补全：从用户表达中提取业务字段，缺失时追问。
4. 流程理解：根据制度文档、操作手册、流程模板和页面结构判断该怎么走。
5. 预检确认：执行前展示任务摘要、字段、风险和缺失项。
6. 系统执行：优先调用 API、MCP 或企业连接器；没有接口时通过 Browser Worker 操作页面。
7. 最终确认：提交、付款、审批、撤回等关键动作前必须确认。
8. Skill 管理：把一个业务流程沉淀为可验证、可发布、可回滚的业务 Skill。
9. 权限控制：继承企业用户身份和权限，不允许 Agent 获得超出用户本人的业务权限。
10. 系统连接管理：展示已连接系统、可用业务 Skill、接口状态和目标系统返回的业务结果。

### 6.2 不做什么

产品明确不做以下事情：

1. 不替代 OA、ERP、CRM、财务系统本身。
2. 不替代 OA 首页或企业门户首页。
3. 不重建企业流程引擎。
4. 不绕过企业已有权限和审批流程。
5. 不保存明文账号密码作为默认方案。
6. 不默认自动提交所有任务。
7. 不把浏览器自动化作为唯一执行方式。
8. 不承诺通过上传操作手册就能可靠执行所有流程。
9. 不在早期支持高风险资金类闭环，例如自动付款、自动对公转账、自动审批付款。
10. 不做面向个人用户的通用“智能按键精灵”作为主线。
11. 不把自学习等同于线上自动改流程。

## 7. 集成策略

### 7.1 P0 入口形态

P0 支持两种入口形态，底层业务执行方式一致，都是 Agent 调用目标系统 API 或连接器。

#### 7.1.1 独立 Agent Workbench

独立 Workbench 适合低成本试点，不要求客户改造 OA 首页。

1. 员工打开 Agent Workbench。
2. Workbench 展示已连接的目标系统，例如 OA、HR、财务。
3. 员工用自然语言发起业务。
4. Agent 调用目标系统 API 或连接器。
5. 目标系统生成草稿、单据号和业务状态。
6. Workbench 展示目标系统返回的结果。

#### 7.1.2 OA 首页悬浮 Widget

如果客户 OA 允许修改首页模板，可以在 OA 首页加入悬浮 AI 办事助手入口。用户点击悬浮入口后，在当前 OA 页面打开聊天小窗口。

示例：

```html
<script src="https://agent.example.com/widget.js" data-tenant="tenant-id"></script>
```

该方式只负责在 OA 首页增加入口和聊天窗口。业务执行仍然依赖 Agent 后端和 OA API，不因为引入脚本而获得业务操作能力。

悬浮 Widget 需要客户配合：

1. 允许 OA 首页加载助手脚本。
2. 提供当前登录用户的短期身份凭证。
3. 配置允许访问 Agent 服务域名。
4. 提供或开放必要业务 API。

如果 OA 不允许自定义脚本，仍可使用独立 Agent Workbench。

### 7.2 同屏展示目标系统

Workbench 可以同屏展示目标 OA 页面或目标系统状态，用于帮助用户理解当前业务结果。该展示不是 P0 的执行依赖。

可选方式：

1. 如果 OA 支持 iframe、SSO 和跨域策略，可在 Workbench 中嵌入 OA 页面。
2. 如果 OA 不支持嵌入，可展示目标系统状态面板，或提供跳转到 OA 原页面的入口。
3. 如果没有 API，后续可使用 Browser Worker 作为兜底，但不作为 P0 首选。

### 7.3 入口集成边界

无论使用独立 Workbench 还是 OA 首页悬浮 Widget，都不替代 OA 本身。入口只解决用户在哪里发起任务，真正的业务能力来自目标系统 API、连接器和业务 Skill。


## 8. 业务工具接入流程

### 8.1 目标

业务工具接入流程面向实施人员、企业 IT 和业务管理员。目标是把企业已有业务系统中的 API、连接器、MCP、页面能力，转成 Agent 可以调用的业务工具，并挂载到对应业务 Skill。

该流程要解决的问题包括：

1. Agent 用什么身份调用目标系统。
2. 企业有哪些业务系统需要接入。
3. 每个业务系统有哪些可用能力。
4. 哪些能力允许开放给 Agent。
5. 如何生成 Agent 可调用的工具定义。
6. 如何验证工具在真实环境或测试环境中可用。
7. 如何把验证后的工具配置到 Agent。

### 8.2 接入流程

业务工具接入流程包括：

1. 配置对接协议和统一用户。
2. 配置业务系统清单。
3. 采集与验证业务系统能力。
4. 手动选择需要开放给 Agent 的业务功能。
5. 根据选取结果生成 MCP、API Tool 或 Browser Skill。
6. 自动配置到 Agent 和对应业务 Skill。
7. 测试 Agent。
8. 验证通过后发布。

### 8.3 配置对接协议和统一用户

实施人员先配置企业身份和系统对接方式。

需要配置：

1. 统一用户来源，例如企业 SSO、LDAP、IAM、OAuth、OIDC。
2. 用户身份映射，例如 Agent 用户与 OA 用户、HR 用户、财务用户的映射关系。
3. 调用授权方式，例如 OAuth Token、API Key、企业网关签名、MCP Server 鉴权。
4. 调用边界，例如只允许以当前用户身份调用，不允许 Agent 获得超出用户本人的业务权限。

### 8.4 配置业务系统清单

实施人员配置本企业要接入的业务系统。

系统清单字段包括：

1. 系统名称，例如 OA、HR、财务、差旅、CRM。
2. 系统类型，例如表单系统、流程系统、查询系统、审批系统。
3. 连接方式，例如 API、MCP、企业连接器、Browser Worker。
4. 环境地址，例如测试环境、预发环境、生产环境。
5. 可用账号或测试身份。
6. 当前接入状态，例如待配置、采集中、待选择、待测试、已发布、已停用。

### 8.5 采集与验证业务系统能力

系统能力来源分为三类：

1. API 文档来源：OpenAPI、Swagger、MCP 描述、厂商 API 文档、企业内部接口文档。
2. 运行环境探测来源：在授权账号或测试账号下自动探测菜单、页面、表单、按钮、校验提示和接口调用结果。
3. 人工录入来源：实施人员手动补充接口、字段、页面规则和异常说明。

不同情况的处理方式：

1. 客户可以提供 API：读取 API 文档，并在测试环境或授权环境中调用验证。
2. 客户无法提供 API：自动探测业务系统页面，识别菜单入口、表单字段、必填项、按钮、校验提示和操作路径，必要时生成 Browser Skill 候选。
3. 客户提供了 API 但需要环境验证：API 调用结果要和页面状态交叉验证，例如草稿是否出现在 OA 页面、提交后状态是否变化、审批链是否正确。

自动采集只用于发现和验证候选能力，不自动把所有能力开放给 Agent。

### 8.6 手动选择业务功能

系统采集完成后，实施人员或业务管理员手动选择需要开放给 Agent 的业务功能。

示例：

1. 请假申请。
2. 假期余额查询。
3. 报销草稿。
4. 工时填报。
5. 审批进度查询。

手动选择的目的是控制 Agent 能力范围，避免把业务系统的全部接口默认暴露给 Agent。

### 8.7 生成工具定义和调用配置

系统根据选取结果生成 Agent 可调用的工具定义。

生成内容包括：

1. 工具名称。
2. 工具描述。
3. 输入参数 schema。
4. 输出结果 schema。
5. 调用方式，例如 MCP、HTTP API、企业连接器、Browser Skill。
6. 鉴权方式。
7. 权限要求。
8. 风险等级。
9. 确认策略。
10. 错误映射。

对于 API 或连接器能力，生成 MCP 或 API Tool 配置。对于没有 API 的页面能力，生成 Browser Skill 候选流程。

### 8.8 配置到 Agent 和业务 Skill

工具定义生成后，系统自动配置到对应 Agent 和业务 Skill。

配置结果包括：

1. 哪个 Agent 可以调用该工具。
2. 哪个业务 Skill 可以使用该工具。
3. 工具在流程中的位置，例如前提条件检查、创建草稿、最终提交。
4. 哪些调用需要用户最终确认。
5. 失败时如何中断或提示用户。

### 8.9 测试和发布

工具配置完成后，必须测试 Agent。

测试用例至少覆盖：

1. 正常路径。
2. 字段缺失。
3. 权限不足。
4. 接口失败。
5. 页面或表单变化。
6. 用户取消。
7. 需要最终确认的写操作。

测试通过后，业务 Skill 才能发布给试点用户使用。

### 8.10 原则

1. API 优先，环境验证必须保留。
2. 无 API 时允许自动探测页面并生成 Browser Skill 候选。
3. 自动采集不等于自动发布。
4. 手动选择决定哪些业务功能开放给 Agent。
5. 测试通过后才能发布。

## 9. 核心业务流程

### 9.1 员工发起任务

员工输入：

> 帮我请明天下午半天年假，原因写个人事务。

Agent 输出结构化理解：

- 任务类型：请假申请
- 假期类型：年假
- 时间：明天下午
- 原因：个人事务
- 风险动作：提交申请
- 确认策略：提交前必须用户确认

### 9.2 Agent 预检

Agent 检查：

- 是否缺少必填字段。
- 是否需要附件。
- 用户是否有权限。
- 假期余额是否满足。
- 是否需要特殊审批。
- 是否存在冲突规则。

预检后展示：

- 将要执行什么。
- 会进入哪个系统。
- 会填写哪些字段。
- 哪些动作需要用户确认。

### 9.3 Agent 执行

执行路径按优先级选择：

1. API / MCP / 企业连接器。
2. 已发布的 Browser Skill。
3. 受控 Browser Worker 流程。

执行过程中必须展示任务状态，并在失败时返回明确错误。

### 9.4 用户最终确认

在提交前，Agent 展示：

- 业务字段摘要。
- 系统页面截图或表单预览。
- 审批人或下一步流转信息。
- 风险提示。

用户确认后才提交。

### 9.5 回执

提交后返回：

- 是否成功。
- 单据号。
- 审批人。
- 当前状态。

## 10. P0 范围

P0 的目标是做出可上线试点的产品闭环，而不是只做演示。

### 10.1 P0 场景

P0 只覆盖低风险、高频、表单明确的内部事务：

1. 请假申请。
2. 加班 / 调休申请。
3. 出差申请草稿。
4. 审批进度查询。
5. 假期余额 / 单据状态查询。

P0 不做完整报销和订票闭环。报销涉及金额、发票、预算、费用科目和规则校验；订票涉及外部供应商、支付、改签、退票和价格变化，复杂度高，不适合作为第一阶段核心范围。

### 10.2 P0 产品能力

1. 自然语言入口：员工可以通过聊天发起 P0 范围内的业务任务。
2. 意图识别：识别 P0 任务类型，并拒绝不支持的任务。
3. 字段抽取：提取任务所需的核心字段。
4. 缺失追问：字段不足时追问，不直接执行。
5. 预检页：执行前展示任务摘要和确认按钮。
6. 用户身份接入：接入统一认证或模拟统一认证，执行权限与用户本人绑定。
7. 执行引擎：至少支持一种确定性执行方式，优先 API 或连接器，没有接口时用 Browser Worker。
8. Browser Worker：支持打开目标系统、填写表单、截图、暂停等待确认、提交。
9. 最终确认：所有写操作提交前必须由用户确认。
10. 任务状态：用户能看到任务处于解析、预检、执行、待确认、提交、完成、失败等状态。
11. 错误处理：支持缺字段、无权限、登录失效、页面变化、表单校验失败、用户取消。
12. 独立 Workbench：通过独立 Agent Workbench 发起业务，不要求改造 OA 首页。
13. OA 首页悬浮 Widget：客户允许修改 OA 首页时，可通过悬浮入口打开聊天小窗口。
14. 目标系统面板：展示已连接 OA、可用 Skill、接口状态和目标系统返回结果。
15. 业务工具接入：支持配置对接协议、业务系统清单、能力采集、人工选择功能、生成工具定义、配置到 Agent。
16. Skill 配置：P0 流程以可配置 Skill 形式存在，不写死在单个脚本里。

### 10.3 P0 管理能力

1. 管理员可以查看已发布的业务 Skill。
2. 管理员可以查看某个 Skill 的适用系统、字段、确认策略和风险等级。
3. 管理员可以禁用某个 Skill。
4. 管理员可以查看失败任务和失败原因。
5. 实施人员可以查看业务系统接入状态。
6. 实施人员可以选择需要开放给 Agent 的业务功能。

### 10.4 P0 成功标准

1. 员工可以在不手动进入业务系统的情况下完成 P0 场景中的至少一个写操作。
2. 写操作提交前一定有用户确认。
3. 任务完成后能返回业务系统生成的结果或单号。
4. 页面变化、字段缺失、权限不足等异常不会静默失败或误提交。
5. P0 不要求替代 OA 首页。
6. 支持独立 Workbench 和 OA 首页悬浮 Widget 两种入口形态。
7. 支持完成至少一个业务系统的工具接入流程，并把选定功能配置到 Agent。

## 11. 后续范围

### 11.1 P1

P1 目标是扩大可用场景，并降低新流程接入成本。

1. 报销草稿，不自动提交付款类动作。
2. 用章申请。
3. 采购申请草稿。
4. 病假附件上传。
5. 多角色权限对比。
6. 流程模板库。
7. 操作手册导入后生成 Skill 草稿。
8. 管理员校正字段映射。
9. Skill 版本管理和回滚。
10. 失败样本归因与修复建议。
11. OA 首页悬浮 Widget 的配置、权限和样式管理。
12. Workbench 中嵌入目标 OA 页面，前提是目标 OA 支持 iframe、SSO 和跨域配置。

### 11.2 P2

P2 目标是从“配置流程”走向“半自动发现流程”。

1. System Mapper：自动发现系统入口、菜单、页面和权限差异。
2. Form Schema Miner：自动识别表单字段、必填项、控件类型和校验规则。
3. Flow Miner：从管理员演示轨迹生成流程草稿。
4. API Miner：从网络请求中发现可复用 API 候选。
5. Permission Profiler：基于不同角色账号生成权限矩阵。
6. Flow Verifier：在 dry-run 或测试环境验证流程。
7. Drift Detector：检测页面、字段、按钮或流程变化。
8. 灰度发布：新 Skill 或新版本先给小范围用户使用。
9. 企业门户、IM 或 OA 插件化入口。

### 11.3 P3

P3 目标是企业级规模化。

1. 多租户隔离。
2. 私有化部署。
3. 复杂组织架构和审批链同步。
4. 更完整的财务、差旅、采购、CRM 场景。
5. 跨系统编排，例如出差申请、订票、报销之间的信息联动。
6. 企业级指标看板，例如节省时间、成功率、失败原因、流程热点。
7. 高风险操作的多级审批策略。
8. 与企业 IM、门户、移动端集成。

## 12. 关键模块

### 12.1 Agent Workbench

员工使用的前台入口。包括聊天、任务卡片、预检确认、执行进度、最终确认、结果回执、已连接系统和目标系统结果展示。

### 12.2 Intent & Field Parser

负责将自然语言解析成业务任务和结构化字段。这个模块只负责理解和抽取，不直接执行。

### 12.3 Policy & Preflight Engine

负责检查字段完整性、权限、风险等级、确认策略、材料要求和业务规则。

### 12.4 Skill Runtime

负责根据任务类型选择对应 Skill，并管理 Skill 的状态、版本、输入输出和执行策略。

### 12.5 Execution Engine

负责真正操作系统。执行方式包括：

1. API / MCP / 企业连接器。
2. Browser Worker。
3. 后续可能支持桌面自动化，但不是主线。

### 12.6 Browser Worker

负责受控浏览器执行。它不应该由模型自由点击，而应该执行已验证的 Skill 步骤，并在关键节点截图、暂停和等待确认。

### 12.7 System Connector Registry

负责管理目标系统连接。包括系统名称、连接方式、可用 API、可用 Skill、身份映射方式、接口状态和结果展示配置。

### 12.8 Tool Onboarding Workbench

面向实施人员和企业 IT。负责配置对接协议、统一用户、业务系统清单，采集与验证业务系统能力，选择要开放给 Agent 的业务功能，并生成 MCP、API Tool 或 Browser Skill 配置。

### 12.9 Lightweight Entry Widget

可选模块。用于在客户 OA 首页增加 AI 办事助手入口。该模块只负责入口和会话启动，不负责替代 OA，也不直接完成业务执行。

### 12.10 Skill Factory

面向管理员和实施人员。负责把企业系统、操作手册、流程模板、页面结构、演示轨迹转成可执行 Skill，并支持验证、发布、回滚。

## 13. 风险与约束

1. 页面自动化脆弱：页面改版会导致执行失败，需要 Drift Detector 和失败降级。
2. 权限边界敏感：Agent 必须继承用户权限，不能拥有额外业务权限。
3. 企业流程复杂：必须把确认、权限边界和失败处理作为产品核心，而不是后补能力。
4. “不用定制”容易过度承诺：对外应表达为低配置、低代码、模板复用、客户私有适配。
5. 自学习不可失控：线上执行逻辑不能由模型自动改写后直接发布。
6. 高风险流程不适合早期覆盖：付款、审批通过、合同签署等动作需要更严格的流程控制。
7. 替代 OA 首页会显著增加实施成本和组织阻力，P0 不走该路线。
8. 一行脚本只能解决入口问题，不能替代身份、权限和业务 API 对接。
9. 自动探测页面和接口只能生成候选能力，不能跳过人工选择和测试发布。

## 14. 产品原则

1. 执行优先：能真正办事，不停留在问答。
2. 可控优先：关键动作必须确认。
3. 低配置优先：把开发变成配置，把配置变成可验证 Skill。
4. API 优先，浏览器兜底。
5. 稳定优先于炫技：P0 不追求自由探索和自进化。
6. 企业边界优先：不绕过身份、权限和审批流程。
7. 不替代入口优先：P0 不替代 OA 首页，入口可以是独立 Workbench 或 OA 首页悬浮 Widget。
8. 工具接入可控：自动采集用于发现和验证，人工选择和测试发布用于控制开放范围。

## 15. 验收标准

P0 验收时，至少满足：

1. 支持一个员工从自然语言发起请假申请并完成提交。
2. 支持一个只读查询场景，例如审批进度或假期余额查询。
3. 支持字段缺失时追问。
4. 支持提交前最终确认。
5. 支持用户取消任务。
6. 支持执行失败并返回明确失败原因。
7. 支持禁用某个业务 Skill。
8. 支持至少 5 类测试用例：正常提交、缺字段、用户取消、权限不足、页面或表单错误。
9. 支持独立 Agent Workbench 入口。
10. 支持 OA 首页悬浮 Widget 入口。
11. 支持展示已连接目标 OA 和业务结果。
12. 支持录入对接协议和统一用户配置。
13. 支持配置业务系统清单。
14. 支持从 API 文档或运行环境采集业务系统能力。
15. 支持人工选择要开放给 Agent 的业务功能。
16. 支持根据选择结果生成 MCP、API Tool 或 Browser Skill 配置。
17. 支持把生成的工具配置到 Agent 并完成测试。
