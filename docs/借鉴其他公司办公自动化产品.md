有，而且我建议不要只学“功能”，而要学它们背后的**产品哲学**。你这个产品最容易走偏成“AI 按键精灵”或“新 RPA”，所以更应该借鉴头部厂商怎么讲 **治理、执行、流程发现、业务对象、Human-in-the-loop、可审计**。

## 1. UiPath：借鉴“Agents think, robots do, people lead”

UiPath 现在的核心叙事不是单纯 RPA，而是 **agentic automation**：AI agents 负责理解上下文和决定下一步，robots 在屏幕、API、应用里执行稳定动作，people 在需要监督时提供方向、审批和专业判断。UiPath 官网直接把平台描述为 “agents think, robots do, people lead”。([UiPath](https://www.uipath.com/?utm_source=chatgpt.com "UiPath: Business Orchestration & Automation Platform")) 它的 Agentic AI 页面也明确说，agentic automation 把 AI agents、robots、integrations 和 people 结合起来，agents 解释上下文，robots 执行高保真动作，people 通过 human-in-the-loop 提供审批和判断。([UiPath](https://www.uipath.com/ai/agentic-ai?utm_source=chatgpt.com "What is Agentic AI? | UiPath"))

**我们应该借鉴：**

|UiPath 思路|对我们产品的启发|
|---|---|
|Agents think|LLM 负责理解用户意图、补字段、判断流程|
|Robots do|Browser Worker / API Connector 负责稳定执行|
|People lead|用户确认、管理员发布 Skill、异常人工接管|
|Orchestration|不只是点页面，而是任务状态机、队列、权限、审计|
|Process Mining / Task Mining|从真实操作中发现流程，而不是完全依赖文档|

**不该照搬：**

不要一开始做成完整 RPA Studio、流程画布、机器人管理平台。那会太重，Demo 和早期销售都很难。

**MVP 落地：**

你可以直接把产品设计成三段式：

```text
AI 规划：理解“帮我请假”
执行器：Browser Worker 填 OA 表单
人确认：用户看摘要和截图后点击提交
```

这比“全自动点击”可信很多。

## 2. Microsoft Copilot Studio：借鉴“低代码 Agent + Connector + Computer Use”

Microsoft Copilot Studio 的定位是：构建和管理 AI agents，连接业务数据，用自然语言创建，并发布到团队和客户使用的渠道。([微软](https://www.microsoft.com/en-us/microsoft-365-copilot/microsoft-copilot-studio?utm_source=chatgpt.com "Microsoft Copilot Studio | Create AI Agents")) 更关键的是，Copilot Studio 已经把 **computer use** 做成 Agent 工具：让 agents 通过图形界面操作 Web 和桌面应用；官方文档说它由 Computer-Using Agents 支持，结合视觉和推理能力与 GUI 交互，并能适应按钮或界面变化。([Microsoft Learn](https://learn.microsoft.com/en-us/microsoft-copilot-studio/computer-use?utm_source=chatgpt.com "Automate web and desktop apps with computer use"))

**我们应该借鉴：**

|Microsoft 思路|对我们产品的启发|
|---|---|
|Agent Builder|需要一个“Skill Builder / Skill Factory”|
|Connectors|能 API 就 API，不要所有东西都用浏览器点|
|Computer Use|没 API 时用 Browser Worker 操作 UI|
|Natural language creation|配置流程时尽量自然语言化，不要全靠代码|
|企业身份体系|SSO / OIDC / SAML / OAuth 是企业信任基础|

**不该照搬：**

不要试图和 Microsoft 拼“通用低代码平台”或 M365 生态。它们优势太强。你的机会是更专注 **非微软、国产 OA、自研系统、内网 Web 系统**。

**MVP 落地：**

第一版可以做两个执行通道：

```text
通道 A：Mock API Connector，演示“查假期余额”
通道 B：Browser Worker，演示“填写请假单”
```

这样客户能理解：你不是只会点页面，也能逐步沉淀 Connector。

## 3. Automation Anywhere：借鉴“Process Reasoning Engine”的产品叙事

Automation Anywhere 现在主打 **Agentic Process Automation**，官网称其是跨团队、系统和流程编排的企业 AI 平台，并强调由 Process Reasoning Engine 驱动。([Automation Anywhere](https://www.automationanywhere.com/home?utm_source=chatgpt.com "Automation Anywhere: The #1 Provider of Agentic Automation")) 它也把 APA 定义为用 AI agents 创建灵活、响应式工作流，完成多步骤、需要规划和决策的流程。([Automation Anywhere](https://www.automationanywhere.com/rpa/agentic-process-automation?utm_source=chatgpt.com "What is Agentic Process Automation? A Complete Guide")) 2025 年发布的信息里还提到其 APA 系统包括 Process Reasoning Engine、Enterprise UI Agents 和 Reasoning AI agents。([Automation Anywhere](https://www.automationanywhere.com/company/press-room/automation-anywhere-takes-step-towards-artificial-general-intelligence-work?utm_source=chatgpt.com "Artificial General Intelligence for Work"))

**我们应该借鉴：**

|Automation Anywhere 思路|对我们产品的启发|
|---|---|
|Process Reasoning Engine|你也需要一个“流程理解/验证引擎”|
|Agentic Process Automation|不要讲“自动填表”，要讲“业务流程自动执行”|
|Enterprise UI Agents|Browser Worker 可以包装成企业 UI Agent|
|多系统编排|请假、出差、报销往往跨 OA/HR/财务/差旅|

**不该照搬：**

不要过早讲“Autonomous Enterprise”。这个词很大，容易让客户质疑落地性。你的早期表达应该更朴素：

> 把一个企业内部 Web 流程，从人工操作变成 AI 预填 + 用户确认 + 可审计执行。

**MVP 落地：**

把你前面说的“自探索、自学习”先包装成：

```text
Process Discovery & Verification Engine
流程发现与验证引擎
```

但 Demo 阶段只展示一个小能力：  
**从请假表单页面抽取字段 schema，并生成可执行步骤。**

## 4. ServiceNow：借鉴“AI Control Tower”的治理视角

ServiceNow AI Control Tower 的核心说法是：发现企业内所有 AI agent、model、identity，治理风险、执行合规、监控运行性能、衡量 AI 价值。([ServiceNow](https://www.servicenow.com/products/ai-control-tower.html?utm_source=chatgpt.com "AI Control Tower")) 它的价值不是“某个 Agent 多聪明”，而是企业能否统一管理这些 AI 能力。

**我们应该借鉴：**

|ServiceNow 思路|对我们产品的启发|
|---|---|
|Control Tower|你也需要一个 Agent / Skill 控制台|
|Discover any AI|发现所有 Skill、任务、执行器、身份|
|Govern risk|每个流程要有风险等级和确认策略|
|Monitor runtime performance|成功率、失败率、耗时、人工接管率|
|Measure value|节省时间、减少返工、自动化完成率|

**不该照搬：**

不要一开始做成完整 AI 治理平台。你的核心还是“企业业务操作 Agent”。治理是信任卖点，不是 MVP 主体。

**MVP 落地：**

做一个很小的 Admin / Audit 页面：

```text
Skill：请假申请
状态：已发布
风险等级：中
确认策略：提交前用户确认
执行次数：12
成功率：91%
最近失败原因：登录超时 1 次
审计记录：可查看截图和步骤
```

这个页面会显著提升企业客户信任感。

## 5. Celonis：借鉴“Process Intelligence / Context Model”

Celonis 的核心是流程智能。它说 process mining 展示的是“流程实际上如何运行，而不是你以为它如何运行”。([celonis.com](https://www.celonis.com/insights/topics/what-is-process-mining?utm_source=chatgpt.com "What is Process Mining? Definition and benefits")) Celonis 也在讲 Enterprise AI 的“Context Model”：结合端到端流程数据、业务知识和智能，让人和 agents 有足够的运营清晰度，能够正确推理、合理决策、可靠行动。([celonis.com](https://www.celonis.com/?utm_source=chatgpt.com "Enterprise AI powered by Celonis"))

**我们应该借鉴：**

|Celonis 思路|对我们产品的启发|
|---|---|
|流程实际如何运行|不能只信操作手册，要看真实轨迹|
|Process Intelligence|先发现流程，再执行流程|
|Context Model|需要企业业务上下文模型，不只是页面 selector|
|流程数据|成功/失败/人工修正都要沉淀|
|端到端视角|报销可能跨出差申请、发票、财务审批|

**不该照搬：**

Celonis 更偏分析和优化。你不要做大型流程挖掘平台，尤其 MVP 不要从 ERP 日志、事件流、流程图开始。

**MVP 落地：**

做一个轻量版 **Capability Graph**：

```text
submit_leave_request
- 入口：OA > 人事 > 请假申请
- 字段：假期类型、开始时间、结束时间、原因
- 风险动作：提交
- 执行方式：Browser Worker
- 确认策略：提交前确认
- 验证状态：verified
```

这就是你未来自探索、自学习、自进化的最小数据结构。

## 6. Salesforce Agentforce：借鉴“业务对象 + 信任层”

Agentforce 的定位是构建和定制 autonomous AI agents，为员工和客户提供 24/7 支持，并且完整集成 Salesforce 生态。([Salesforce](https://www.salesforce.com/ap/agentforce/?utm_source=chatgpt.com "Agentforce: The AI Agent Platform")) Salesforce 也强调 Agentforce 可以连接企业数据，并在销售、服务、营销和商务流程中采取行动。([Salesforce](https://www.salesforce.com/news/press-releases/2024/10/29/agentforce-general-availability-announcement/?utm_source=chatgpt.com "Salesforce's Agentforce Is Here: Trusted, Autonomous AI ..."))

**我们应该借鉴：**

|Salesforce 思路|对我们产品的启发|
|---|---|
|Agent grounded in business data|Agent 必须基于业务对象和企业数据，不只是聊天|
|Actions across business processes|Agent 要能执行动作，而不是只回答|
|Trust|企业 Agent 必须有权限、数据边界、审计|
|Role-based agent|不同岗位有不同 Agent Skill|
|CRM object model|你也要抽象 OA/HR/财务对象模型|

**不该照搬：**

不要像 Salesforce 一样强绑定某个业务系统生态。你的差异反而是跨系统：OA、ERP、HR、财务、CRM、自研系统都能接。

**MVP 落地：**

不要把请假看成“网页表单”，要抽象成业务对象：

```text
LeaveRequest
- applicant
- leave_type
- start_time
- end_time
- reason
- attachment
- approver
- status
```

这样未来无论底层是 OA 页面、API、飞书、钉钉还是自研系统，都能映射到同一个业务对象。

## 7. Blue Prism：借鉴“治理、合规、变更管理”

SS&C Blue Prism 自己强调它面向复杂业务，提供 trusted、tested、ready to run 的 agentic automation。([SS&C Blue Prism](https://www.blueprism.com/?utm_source=chatgpt.com "Blue Prism")) 它的智能自动化指南也提醒，IA 会带来治理、合规、数据质量、安全和变更管理挑战；治理不足会导致结果不一致，安全缺口和流程快速变化会增加运营风险。([SS&C Blue Prism](https://www.blueprism.com/guides/intelligent-automation/?utm_source=chatgpt.com "What is Intelligent Automation?")) Blue Prism 还强调 RPA governance framework 能确保自动化流程保持合规、符合业务目标并持续高效运行。([SS&C Blue Prism](https://www.blueprism.com/automation-journey/rpa-governance/?utm_source=chatgpt.com "IA & RPA Governance Best Practices"))

**我们应该借鉴：**

|Blue Prism 思路|对我们产品的启发|
|---|---|
|Governance-first|企业客户先信任，再自动化|
|Change management|OA 页面变了要检测、降级、回滚|
|Compliance|操作日志、确认记录、截图留存|
|CoE / 管理机制|客户管理员要能审批和发布 Skill|
|Tested automation|Skill 必须有验证状态，不是模型自由发挥|

**不该照搬：**

Blue Prism 偏传统企业 RPA，实施和治理很重。你不要让早期产品变成咨询项目。

**MVP 落地：**

每个 Skill 增加几个状态：

```text
Draft → Verified → Published → Degraded → Disabled
```

页面变化或失败率上升时自动从 Published 降级到 Degraded，不再自动提交，只允许预填或转人工确认。

## 8. Browserbase / Browser Use：借鉴“远程浏览器基础设施 + Live View”

Browserbase 的定位是给 agents 提供浏览器基础设施，官网强调可以启动上千个并发浏览器会话；也提到可用于持续点击产品流程并在流程坏掉时告警。([Browserbase](https://www.browserbase.com/?utm_source=chatgpt.com "Browserbase")) 它的 Live View 文档支持获取浏览器 session 的实时调试链接。([Browserbase 文档](https://docs.browserbase.com/platform/browser/observability/session-live-view?utm_source=chatgpt.com "Session live view - Browserbase Documentation"))

**我们应该借鉴：**

|Browserbase 思路|对我们产品的启发|
|---|---|
|Session abstraction|每次任务是一个 Browser Session|
|Concurrent sessions|Browser Worker Pool 要能多并发|
|Live View|用户/管理员可以看到 Agent 正在操作|
|Debugging|出错时能回放、定位、修复|
|Broken flow detection|定期跑流程探针，发现页面变化|

**不该照搬：**

Browserbase 是基础设施。你不能只做“浏览器云服务”，否则容易被替代。你的价值在上层：

```text
业务 Skill + 权限 + 预检 + 确认 + 审计 + 企业系统适配
```

**MVP 落地：**

哪怕不做真正实时 Live View，也要做：

```text
步骤截图
当前状态
失败截图
执行 trace
用户确认页
```

这就是你的轻量 Live View。

## 9. OpenAI / Anthropic / Microsoft Computer Use：借鉴“风险点确认”

OpenAI Computer Use 文档强调，在高影响动作、提交敏感数据、外部副作用动作前，要让用户确认。([Microsoft Learn](https://learn.microsoft.com/en-us/microsoft-copilot-studio/computer-use?utm_source=chatgpt.com "Automate web and desktop apps with computer use")) OpenAI 的 CUA / Operator 路线也体现了：模型可以自动执行很多浏览器步骤，但登录、验证码、敏感操作需要用户介入或确认。Anthropic 也把 computer use 定位为让 AI 使用软件界面，而不是依赖专门 API。([Automation Anywhere](https://www.automationanywhere.com/company/press-room/automation-anywhere-takes-step-towards-artificial-general-intelligence-work?utm_source=chatgpt.com "Artificial General Intelligence for Work"))

**我们应该借鉴：**

|Computer Use 思路|对我们产品的启发|
|---|---|
|GUI 操作是可行路径|没 API 的 OA 也能做|
|视觉 + DOM + 推理|不要只靠 selector，也不要只靠截图|
|Human-in-the-loop|登录、验证码、提交、付款、审批必须确认|
|Risk-based policy|不同操作不同确认等级|
|Session cleanup|任务结束清理 cookie、文件、截图缓存|

**不该照搬：**

不要让模型直接自由控制浏览器完成企业流程。企业 Demo 和生产环境应尽量是：

```text
LLM 规划 + 状态机约束 + Playwright 执行 + 用户确认
```

而不是：

```text
LLM 想点哪里就点哪里
```

# 对我们产品最重要的 10 个可借鉴设计

## 1. Agent / Worker / Human 三权分立

来自 UiPath。

```text
Agent：理解和规划
Worker：执行和取证
Human：确认和审批
```

这能防止产品被认为是危险的黑盒自动化。

---

## 2. Skill Factory，而不是单个 Agent

来自 Microsoft / Automation Anywhere / ServiceNow 的 Agent Builder 思路。

你的核心不是“一个万能 Agent”，而是：

```text
把每个企业流程转成可发布、可验证、可回滚的 Skill。
```

例如：

```text
submit_leave_request
create_expense_draft
query_approval_status
create_travel_request
```

---

## 3. Capability Graph / Context Model

来自 Celonis + Salesforce。

不要只存 Playwright 脚本，要存业务能力：

```text
业务对象
字段 schema
流程步骤
权限约束
风险动作
执行方式
验证状态
```

这是未来自探索、自学习、自进化的基础。

---

## 4. API 优先，Browser Worker 兜底

来自 Microsoft Connector + Computer Use。

```text
有 API：走 Connector
无 API：走 Browser Worker
API 不稳定：UI fallback
UI 稳定后：反向生成 API 候选
```

这个策略比纯 Browser Worker 更企业级。

---

## 5. Preflight / 材料包

这是你自己前面提到的方向，也能和 RPA 的 attended/unattended 思路结合。

```text
先收集材料
先补字段
先确认授权范围
再启动 Browser Worker
最终提交前再确认
```

这样减少用户中途被打断，也减少浏览器资源占用。

---

## 6. AI Control Tower / Admin Console

来自 ServiceNow / Blue Prism。

最小也要有：

```text
Skill 列表
任务记录
执行状态
成功率
失败原因
审计截图
风险等级
确认策略
启用/禁用
版本回滚
```

这对企业销售非常重要。

---

## 7. Risk-based Confirmation

来自 OpenAI / Microsoft Computer Use / 企业 RPA 治理。

不要所有动作都一样：

|动作|策略|
|---|---|
|查询|可自动|
|填草稿|可自动|
|上传附件|用户预授权后自动|
|提交申请|提交前确认|
|审批同意|二次确认|
|付款/权限变更|默认禁止或强审批|

---

## 8. Live View / Screenshot Review

来自 Browserbase。

企业客户不一定需要实时看全程，但至少要看到：

```text
Agent 当前做了什么
填了哪些字段
页面截图是什么
即将提交什么
失败时卡在哪里
```

这会极大提升信任。

---

## 9. Drift Detection / Broken Flow Detection

来自 Browserbase 的“流程坏掉前发现”和 Blue Prism 的变更管理。

企业 OA 页面会变。你的系统要能检测：

```text
selector 失效
字段消失
按钮文案变化
页面跳转异常
错误提示新增
成功率下降
```

发现后自动降级：

```text
自动提交 → 只预填
只预填 → 转人工
Published → Degraded
```

---

## 10. Skill Versioning / 灰度 / 回滚

来自企业软件治理。

每个流程都要像软件版本一样管理：

```text
leave_request v1.0
leave_request v1.1 修复字段识别
leave_request v1.2 增加病假附件
```

发布流程：

```text
候选变更 → dry-run → 管理员审核 → 小范围灰度 → 全量发布 → 可回滚
```

# 对我们产品的最终建议

你这个产品最应该借鉴的不是某一家，而是组合成一个新形态：

```text
UiPath 的执行分工
+ Microsoft 的 Connector / Computer Use
+ Browserbase 的远程浏览器 Session
+ Celonis 的流程上下文模型
+ ServiceNow 的治理控制台
+ Blue Prism 的合规和变更管理
+ Salesforce 的业务对象和信任层
```

最终产品形态可以这样表达：

> **一个面向企业内部 Web 系统的 Agent Skill Factory。它通过自然语言理解业务意图，通过 API Connector 或 Browser Worker 执行，通过 Preflight 收集材料，通过 Human-in-the-loop 控制风险，通过 Capability Graph 沉淀流程能力，并通过 Control Tower 管理权限、审计、版本和回滚。**

对 Demo 来说，不要全都做。Demo 只做四个最关键借鉴：

```text
1. UiPath：Agent 规划、Worker 执行、人确认
2. Microsoft：Browser Worker 操作无 API 页面
3. Browserbase：截图/Live View 展示执行过程
4. Blue Prism：审计记录和风险确认
```

这样 Demo 会很聚焦，也能讲出企业级产品的味道。
